## 1. 레이어드 렌더링(Layered Rendering) 구조로의 전환

기존에는 UI 클래스의 `Render` 함수가 화면 비우기부터 출력까지 모든 과정을 처리하여, 씬 위에 다른 UI를 겹쳐 그리거나 씬 고유의 그래픽(아스키 아트 등)을 삽입하기 어려웠습니다. 이를 해결하기 위해 렌더링 파이프라인을 분리했습니다.

* **개선된 렌더링 파이프라인**: `ClearBuffer` (화면 비우기) → **씬/로컬 UI 렌더링** → **전역 UI 렌더링** → `Draw` (화면 출력) 순으로 진행됩니다.
* **역할 분담**: 배경이나 캐릭터 등 매 프레임 좌표 기반으로 그려야 하는 요소는 씬의 `Render()`에서 담당하고, 텍스트 로그는 기존의 `AddMessage` 시스템을 유지하여 효율성을 높였습니다.

## 2. UI 관리 주체 분리 (Global vs Local)

`UIManager`가 모든 UI를 관리하던 구조에서, 전역적으로 유지되는 UI와 특정 씬에서만 쓰이는 UI를 분리하여 데이터 독립성을 확보했습니다.

* **Global UI**: 게임 전반에 걸쳐 유지되는 로그(Log)와 하단 메시지 창은 `UIManager`가 싱글톤으로 관리합니다.
* **Local UI**: 미니맵, 캐릭터 상태창 등 특정 씬에만 종속된 UI는 각 씬(`BaseScene`)이 소유하는 `ui_list`에서 관리하며, 씬이 파괴될 때 함께 메모리에서 해제됩니다.

## 3. 씬 불투명도(Opaque)를 통한 스택 렌더링 최적화

씬 스택에 쌓인 모든 씬을 매번 그리는 것은 비효율적이며, 전투 씬처럼 아래 화면을 완전히 가려야 하는 경우 잔상이 남는 문제가 있었습니다.

* **불투명도 속성 도입**: `BaseScene`에 `is_opaque` 플래그를 추가했습니다.
* **렌더링 최적화**: `GameManager`는 렌더링 시 스택의 최상단부터 아래로 내려가며 처음 만나는 불투명한 씬까지만 찾습니다. 그 인덱스부터 최상단까지만 그림으로써 불필요한 하위 씬의 렌더링을 차단합니다.
    * **전투 씬**: 불투명 속성을 부여하여 아래의 던전 화면이 보이지 않게 처리합니다.
    * **인벤토리(예정)**: 투명 속성을 부여하여 아래 씬이 비쳐 보이도록 연출할 수 있습니다.

## 4. 오브젝트 풀링과 전투 시스템

무작위 생성형 던전에서 빈번하게 발생하는 몬스터 생성/삭제의 메모리 오버헤드를 줄였고, 기본 전투 로직을 구현했습니다.

* **몬스터 오브젝트 풀링 (`MonsterPool`)**:
    * **재사용**: `Monster` 클래스에 기본 생성자와 내용을 덮어쓰는 `Spawn()` 함수를 추가했습니다.
    * **통제**: `Init()`을 통해 게임 시작 시점에 메모리를 미리 할당하여 프레임 드랍을 방지하고 초기화 시점을 통제합니다.
    * **상태 관리**: `is_visible` 플래그를 통해 풀 내의 유휴 객체를 찾아 활성화합니다.
* **전투 매니저 (`BattleManager`)**:
    * **의존성 분리**: 플레이어와 몬스터의 포인터를 참조하여 전투 공식을 처리하며, 싱글톤이 아닌 `GameManager` 소유 방식으로 설계하였습니다
